## Background

We are given a netcat connection that first gives us three values: a nonce, the encrypted flag, and an authentication tag for the flag. Then, we can request for a number of additional ciphertexts. Each request gives us a new nonce, a new ciphertext, and a new tag. The encryption algorithm for the flag is standard [AES-GCM](https://en.wikipedia.org/wiki/Galois/Counter_Mode) (see `encrypt_flag_message`). When requesting additional ciphertexts, as seen in `get_tag`, we see that the input messages are securely randomly generated. Then, some encryption algorithm that _looks_ like AES-GCM encrypts the message and produces a tag. Also, we only have a query budget of 704.

### Faulty AES-GCM

Now, I analyzed the code used to encrypt the queries. There are a lot of functions and I was not entirely sure what the standard AES-GCM implementation is. I asked (insert your favorite LLM) which of the functions in the code are identical to an analogous function in AES-GCM and which functions have been modified.

> Focus on the custom fault-injection logic in `construct_tag`, `encrypt_query_message`, `next_fault_words`, `encrypt_query_with_rng`, `get_tag`, and `main` for the vulnerability.

So these functions are the ones that are not standard AES-GCM. Note that `encrypt_query_with_rng`, called from `get_tag`, is used to encrypt and authenticate a provided message, so it is the top-most function we have to analyze. This calls `next_fault_words` and then uses these in a call to `encrypt_query_message`. `next_fault_words` generates two random 32-bit integers. `encrypt_query_message` _encrypts_ the message normally. It then computes a 128-bit value called `ghash` then calls `construct_tag`. This is where the code is interesting.

`construct_tag` essentially XORs the lower 64 bits of ghash with the two fault words (which are combined into a 64-bit integer). Very interesting. For context, in normal AES-GCM, the ghash should be "encrypted" with AES-GCM (i.e. XORed with the AES encryption of the IV/nonce + some padding). The keystream to XOR the ghash with is computed with `compute_gcm_mask` as seen in the code.

Also, the random numbers generated by `next_fault_words` use Python's `random.getrandbits` function, which is not cryptographically secure! This will be very important.

## What's the Issue

### ghash Calculation

In this challenge, only one block is encrypted at a time. This is interesting when we break down the ghash calculation (see `compute_one_block_ghash`). There is an int h which is derived from a master key (see `compute_h` - it is the AES encryption using the master key of a block of 0-bytes). Then, the function `ghash` is called to compute the ghash over two blocks: the ciphertext block and a "length" block which encodes the length of the ciphertext (see `compute_length_block`). Inside `ghash`, we call `gf_mul` for each block, which works in the field GF(2^128). Let `L` be the 128-bit encoded length and `C` be the 128-bit ciphertext as GF(2^128) elements. Then, `ghash = C*h^2 + L*h`.

### The Vulnerability

Note that in a field of characteristic 2 (such as GF(2^128)), squaring is linear. That is, `(a+b)^2 = a^2 + b^2`. Therefore, `ghash` is simply a linear function of h. This makes just the value of `ghash` insecure as a tag because with enough tag observations, you can recover h. This is why in AES-GCM, `ghash` is "encrypted".

However, we don't get values of `ghash` - the lower 64 bits are XORed with a random fault. Moreover, even if we can recover h, we can only forge new authentication tags, not decrypt the original flag.

But notice that our "master key" seeds the RNG used to create the faults! With enough observations of fault values, we may be able to solve for the original state of the RNG, and recover the master key, allowing us to decrypt the flag!

## Recovering the Mersenne Twister State

Crucially, the RNG used to generate faults is Python's random module, which uses the Mersenne Twister. This RNG stores an internal state of 624 32-bit words (19968 bits). Each time it generates a number, a certain known linear function ($f_i$) of these state bits is computed.

Each time we query a ciphertext ($c_i$), we get to see its "fake" tag. For tag $t_i$, working in vectors over GF(2), we have $t_i = ghash_i + fault_i = c_ih^2 + lh + f_i(s)$. Note that this is not exactly right, since only the lower 64 bits of $c_i$ are useful to us. Here, $s$ is the initial 19968-bit Mersenne Twister state vector, $l$ is the constant 128-bit encoded length, and $h$ is a 128-bit vector.

Since squaring is linear in GF(2^128), we can find a matrix $S$ such that $S\vec{h}$ = $\vec{h^2}$ if we write $h$ as a vector $\vec{h}$ in GF(2). The following code in Sage does this:

```py
K.<a> = GF(2^128)
V, from_V, to_V = K.vector_space()
# column j is the vector for the polynomial a^2j
S = matrix(GF(2), [to_V(a^(2*j)) for j in range(128)]).transpose()
```

Likewise, we also have to construct matrices that do scalar multiplication in GF(2^128) by both $C_i$ and $L$. The following Sage code does this:

```py
K.<a> = GF(2^128)
V, from_V, to_V = K.vector_space()

# x is a GF(2^128) polynomial element
def mul_matrix(x):
    return matrix(GF(2), [to_V(x * a^j) for j in range(128)]).transpose()

# convert an int n to a GF(2^128) polynomial element, read the bits in the same order that `gf_mul` does so
def from_int(n):
    return K([(n >> (127 - i)) & 1 for i in range(128)])

# opposite of from_int
def to_int(el):
    v = to_V(el)
    return sum(int(v[i]) << (127 - i) for i in range(128))

def mul_matrix_from_int(n):
    return mul_matrix(from_int(n))
```

Finally, we need a matrix $B_i$ that represents the linear transformation $f_i$. I used [this repository](https://github.com/obafgkmdh/randcrack-linear) by [clam](https://github.com/obafgkmdh/) to generate the 64 column vectors in this matrix that we use for each observation:

```py
    word0 = sym_rng.genrand_uint32()  # list of 32 vectors in GF(2)^19968
    word1 = sym_rng.genrand_uint32()  # list of 32 vectors in GF(2)^19968
```

Putting it all together, suppose we observe a ciphertext $c_i$. Then, let $C_i = \text{mul_matrix_from_int}(c_i)$. $l$ is the 128-bit constant bit length of a block (0x80), and let $L = \text{mul_matrix_from_int}(L)$. Then, $A_i = C_i * S + L$, which is a 128x128 square matrix. Note, however, that the fault only applies to the lower 64 bits of the ghash/tag and thus that is the only part useful to us. Let ${A_i}'$ be the bottom half of $A_i$, which is a 64x128 matrix. Then, $B_i$ is the 64x19968 matrix operating on the Mersenne Twister state. Let $M_i = A_i \mid B_i$, which is a 64x20086 matrix representing the linear equation that we observe on a tag for each query.

There are 20086 unknowns (19968 bit initial Mersenne Twister state + 128-bit h, which does not change). We get 64 "equations" for each query. To solve the system, we therefore need at least 314 equations, which is well under our query budget of 704. We can simply make all these queries, gather the tag and ciphertext values we see, create a matrix $M$ by stacking each $M_i$ that we construct, and solve the system $M\vec{x} = \vec{y}$, where $\vec{x}$ are the unknowns and $\vec{y}$ are the observed tag values (we only save the lower 64 bits of each tag for our observations).

## Recovering the Key

We have done most of the hard work of this challenge, but we still don't actually have the master key yet! We do have the correct 624-bit initial Mersenne Twister state which was seeded by the master key. How do we reverse engineer the master key from this state?

The master key is 32 random bytes and we seem to seed the RNG by converting it to a big-endian integer. How does Python seed the RNG in this case? We need to dig a big deep into the CPython internals for `random.seed`.

### Random seeding

We need to look at [`_randommodule.c`](https://github.com/python/cpython/blob/09233bd19879284395aff97d7357b693893e6dd7/Modules/_randommodule.c) to understand random seeding. The important function is `random_seed`. Since we are seeding with an integer, we can skip most of the setup code and go here:

```c
    /* Now split n into 32-bit chunks, from the right. */
    bits = _PyLong_NumBits(n);
    assert(bits >= 0);
    assert(!PyErr_Occurred());

    /* Figure out how many 32-bit chunks this gives us. */
    keyused = bits == 0 ? 1 : (size_t)((bits - 1) / 32 + 1);

    /* Convert seed to byte sequence. */
    key = (uint32_t *)PyMem_Malloc((size_t)4 * keyused);
    if (key == NULL) {
        PyErr_NoMemory();
        goto Done;
    }
    res = _PyLong_AsByteArray((PyLongObject *)n,
                              (unsigned char *)key, keyused * 4,
                              PY_LITTLE_ENDIAN,
                              0, /* unsigned */
                              1); /* with exceptions */
    if (res == -1) {
        goto Done;
    }
```

This simply splits up our large integer into 32-bit integers and places them into an array. The first element is the least significant 32 bits of the large integer, and so on. We can rewrite this behavior in Python:

```py
MASK32 = (1 << 32) - 1
def convert_to_array(seed_int):
    n = seed_int
    out = []
    while n:
        num = n & MASK32
        out.append(num)
        n >>= 32
    return out
```

Then, `random_seed` calls `init_by_array`, which does most of the work in seeding the initial Mersenne Twister State. Let's look inside the function.

```c
/* Period parameters -- These are all magic.  Don't change. */
#define N 624
#define M 397
#define MATRIX_A 0x9908b0dfU    /* constant vector a */
#define UPPER_MASK 0x80000000U  /* most significant w-r bits */
#define LOWER_MASK 0x7fffffffU  /* least significant r bits */
static void
init_by_array(RandomObject *self, uint32_t init_key[], size_t key_length)
{
    size_t i, j, k;       /* was signed in the original code. RDH 12/16/2002 */
    uint32_t *mt;

    mt = self->state;
    init_genrand(self, 19650218U);
    i=1; j=0;
    k = (N>key_length ? N : key_length);
    for (; k; k--) {
        mt[i] = (mt[i] ^ ((mt[i-1] ^ (mt[i-1] >> 30)) * 1664525U))
                 + init_key[j] + (uint32_t)j; /* non linear */
        i++; j++;
        if (i>=N) { mt[0] = mt[N-1]; i=1; }
        if (j>=key_length) j=0;
    }
    for (k=N-1; k; k--) {
        mt[i] = (mt[i] ^ ((mt[i-1] ^ (mt[i-1] >> 30)) * 1566083941U))
                 - (uint32_t)i; /* non linear */
        i++;
        if (i>=N) { mt[0] = mt[N-1]; i=1; }
    }

    mt[0] = 0x80000000U; /* MSB is 1; assuring non-zero initial array */
}
```

`mt` is an array of 624 32-bit words (the Mersenne Twister state). First, let's look at `init_genrand`.

```c
/* initializes mt[N] with a seed */
static void
init_genrand(RandomObject *self, uint32_t s)
{
    int mti;
    uint32_t *mt;

    mt = self->state;
    mt[0]= s;
    for (mti=1; mti<N; mti++) {
        mt[mti] =
        (1812433253U * (mt[mti-1] ^ (mt[mti-1] >> 30)) + mti);
        /* See Knuth TAOCP Vol2. 3rd Ed. P.106 for multiplier. */
        /* In the previous versions, MSBs of the seed affect   */
        /* only MSBs of the array mt[].                                */
        /* 2002/01/09 modified by Makoto Matsumoto                     */
    }
    self->index = mti;
    return;
}
```

This just seeds initial values into the state depending on `s`. We can rewrite it in Python:

```py
MASK32 = (1 << 32) - 1
def init_genrand(s, mt_state):
    mt_state[0] = s
    for i in range(1, N):
        mt_state[i] = (1812433253 * (mt_state[i-1] ^ (mt_state[i-1] >> 30)) + i) & MASK32
```

Then, `init_by_array` has 2 loops where it initializes the contents of the state according to the provided key array. In our case, since we split up the 256-bit key into an array of 32-bit integers, there are 8 elements in this array.

### Reversing init_by_array

How do we reverse `init_by_array` to recover the key? My first thought was to use Z3, but with 256-bit bitvectors with so many constraints, it will likely take too long to run. Crucially, however, most steps in the seeding algorithm are reversible! First, note that i starts at 1 and ends at 2 after the first loop, where it wraps around N=624 once. Then, i starts at 2 and ends at 2 after the second loop, where it also wraps around N=624 once. Let's start with the second loop:

```c
    for (k=N-1; k; k--) {
        mt[i] = (mt[i] ^ ((mt[i-1] ^ (mt[i-1] >> 30)) * 1566083941U))
                 - (uint32_t)i; /* non linear */
        i++;
        if (i>=N) { mt[0] = mt[N-1]; i=1; }
    }

    mt[0] = 0x80000000U; /* MSB is 1; assuring non-zero initial array */
```

First of all, we know the old value of `mt[0]` before the last line: it is `mt[N-1]` since in the loop, i reaches N. Then, notice that during the forward iteration of the loop, `mt[i]` depends on `mt[i-1]`, which was already set by the loop. Therefore, we can simply reverse the loop by looking at `mt[i-1]` as this is the same value seen in the forward iteration of the loop. I will spare some of the details when reversing the loop because they aren't that interesting honestly (and a LLM would probably have an easier time doing this than me). But here is the code to reverse the second loop:

```py

        if i == 0:
            # need previous value of state[0] which is only known from previous value of state[N-1] (this is set in the first loop!)
            i = N-1
	...
```

Reversing the first loop is somewhat similar. The same pattern exists where `mt[i-1]` is the same value used on both the forward and backward iterations of the loop. However, now, we are trying to recover the key. This means we need to compare the "old" value in the Mersenne Twister state and the "new" version, and their difference can be used to derive one of the 32-bit key words. The "old" value in the state is (usually) the one generated by the `init_genrand` call. The one exception is when i=1 because the first loop crosses i=1 twice, so the value we see at i=1 has been twice overwritten. Also, we only need to reverse this loop enough times to fill in the 8 words of our key, which means we only need to recover 8 values. Again, I am sparing some details, so here is the full code for `reverse_init_by_array`:


### Untwisting

Notice that in `init_by_array`, we see `mt[0] = 0x80000000U; /* MSB is 1; assuring non-zero initial array */` in the last line. For a sanity check, let's make sure that the recovered Mersenne Twister state from the sage script has the same first element. But... it is different? Why could that be?

Notice this line in `init_genrand`: `self->index = mti;`. This effectively sets the index of the random object to 624 (or N). What happens when we generate the first random number? `genrand_uint32` generates random 32-bit numbers, and if the index is N, it re-twists the Mersenne Twister state! So that means after seeding our RNG, we also have to account for an initial twist of the Mersenne Twister. Great... more things to reverse engineer.

Actually not, as we can take a little shortcut. Twisting the Mersenne Twister state happens every 624 calls to `genrand_uint32`. What if we just encode these calls into the vectors we used to solve for the initial state? In Sage, before generating symbolic vectors that we actually use, we can just generate 624 initial symbolic vectors and throw them away to advance the symbolic state 624 times and trigger a symbolic twist. That's pretty easy!

```py
# model a twist by querying RNG 624 times
for _ in range(624):
	sym_rng.genrand_uint32()
```

## Finishing the solve

With these steps, we can now solve for the initial, untwisted Mersenne Twister state. We also can reverse this back into the master key used to seed the RNG. All that's left is to decrypt the flag using the nonce and ciphertext we received, and we have our flag! Here is the full solve script:
